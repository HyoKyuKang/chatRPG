이번 worker = step 1 (분석 + lint 박기). step 2 (사용자 우선순위) + step 3

이어서).

### Step 1.A — scripts/lint-balance.ts 박기 (W3 lint-echoes 패턴)

`scripts/lint-balance.ts` 작성:
- 모든 노드 graph walk (loadNodes / loadKnowledge 활용)
- 다음 패턴 catch 후 warn:
  - **자기 회복 무한 loop:** type='event' / 'discovery' 노드 가 stat-gain
    knowledge 또는 effects 박혔는데 visited gate (예: knowledge='rested-at-X')
    없이 self-reachable path (graph 의 cycle 또는 backtrack)
  - **shop / 아이템 받기 무한:** 같은 노드 의 itemGain 효과 가 visited gate
    없이 반복 가능
  - **보스 우회 abuse:** boss 노드 의 nextNodeId 가 회피 path 통해 다시 보스
    재진입 가능 + 보상 재수여
  - **메타 진척 abuse:** memory shard / unlock 효과 가 한 출정 안에 multiply
    trigger 가능
- 출력: 의심 spot list (노드 ID + 패턴 종류 + 미충족 gate 추천)
- 0 warn = clean (exit 0), N warn = list (exit 0, fail X)
- npm run lint:balance 명령 추가 (package.json scripts)

### Step 1.B — manual review (lint 못 잡는 거)

`notes/balance-sweep-manual.md` 작성:
- lint 결과 + manual catch (예: 영웅 영입 의도 깨짐 — 한 출정에서 12 영웅 다
  영입 가능?)
- 우선순위 매김 (HIGH = 출시 차단 가능, MED = UX 부담, LOW = nice-to-fix)
- 각 spot 별 fix 추천 (data 갱신 / schema 보완 / store logic)

### Step 1.C — 사용자 결정용 보고

위 두 결과 합쳐서 한 줄 + list 보고. 사용자가 우선순위 결정.

## 제약
